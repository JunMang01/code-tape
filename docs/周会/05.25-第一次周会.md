## 2. 需要老师重点确认的 5 个问题

### 2.1 P0 范围：是否继续收缩

我当前建议 P0 只保住这条闭环：

> 编辑器写代码 -> 采集操作事件和音视频 -> 生成录制包 -> 本地保存 -> 加载回放 -> 支持播放、暂停、倍速、seek、音量/静音。

P0 必须做：

- JS/TS 编辑、高亮、基础补全。
- 内容变更、光标选区、鼠标轨迹、快捷键采集。
- 麦克风和摄像头采集，摄像头悬浮预览。
- 开始、暂停、继续、结束录制。
- 本地录制包保存和加载。
- 回放时还原编辑器状态、鼠标、选区、快捷键、运行结果和摄像头视频。

P0 不建议做：

- 后端 Docker / VM 沙箱。
- WebContainers、npm install、终端、多文件工程运行。
- WebRTC 实时面试。
- 完整 AI 字幕生成和纠错。
- 账号、权限、云端分享。

想请老师判断：

1. 这个 P0 范围是否已经足够收敛？
2. 是否有被我砍掉的能力其实会影响验收？
3. 如果后续时间再压缩，应该优先砍“媒体同步细节”“导入导出”“TS 简单运行”里的哪一项？

### 2.2 技术路线：是否认可事件流优先

我当前技术路线可以概括为：

> 不录屏，而是记录带时间戳的结构化事件；回放时用统一时间轴调度事件和媒体。

关键取舍：

| 方向 | 当前方案 | 原因 |
| --- | --- | --- |
| 编辑器 | Monaco Editor | JS/TS 支持成熟，方便采集内容、选区、快捷键 |
| 代码运行 | iframe sandbox | P0 成本最低，符合 PRD 中“前端代码页面展示”路线 |
| 内容变更 | 保存完整代码快照 | 降低回放 seek 和状态恢复复杂度 |
| 录制包 | IndexedDB + zip 兜底 | 先完成本地闭环，云端放 P1 |
| 回放 seek | 最近快照 + 增量事件静默重放 | 避免每次 seek 从头重放 |
| 媒体同步 | 单 WebM + MediaClockAdapter | 降低音视频和事件流同步复杂度 |
| 回放运行结果 | 展示历史结果，不默认重新执行 | 避免回放时引入不确定性和安全风险 |

想请老师判断：

1. “结构化事件流”作为产品主线是否正确？
2. iframe sandbox 作为 P0 运行路线是否足够支撑验收？
3. 完整代码快照是否可以接受，还是必须更早做 diff/operation log？
4. seek 的“快照 + 增量重放”是否需要提前做专项压测？

### 2.3 Demo 口径：突出结构化回放，还是在线运行效果

我当前建议 Demo 叙事是：

> code-tape 不是普通录屏工具，而是代码讲解的结构化时间线。它能记录写代码、讲解、运行、移动鼠标、选择文本、触发快捷键和摄像头讲解，并在回放时支持 seek 和倍速。

建议 Demo 展示顺序：

1. 打开录制工作台。
2. 在 Monaco 中写一段 JS/TS 代码。
3. 点击运行，iframe 输出结果，并把运行结果写入事件流。
4. 开始录制，展示计时、鼠标、光标选区、快捷键采集。
5. 开启麦克风和摄像头，展示悬浮预览。
6. 暂停/继续录制，说明暂停期间不产生不可解释的状态跳变。
7. 停止录制，生成并保存录制包。
8. 从本地回放中心进入回放。
9. 在回放中展示代码变化、运行结果、鼠标激光笔、快捷键浮层、摄像头视频。
10. 演示播放/暂停、倍速、seek、音量和静音。

当前预定演示环境：

- 桌面 Chrome。
- 部署地址：https://ceilf6.github.io/code-tape/
- 单次演示录制控制在 5 分钟以内。
- 单文件代码控制在 200KB 以内。
- 事件数量控制在 20,000 条以内。
- media Blob 控制在 200MB 以内。
- seek 后 300ms 内恢复编辑器稳定状态。
- 媒体漂移目标小于 250ms。

想请老师判断：

1. 最终展示应更强调“结构化回放体验”，还是“在线运行代码效果”？
2. 如果现场媒体权限或浏览器兼容出问题，是否可以用备用录制包兜底？
3. 演示环境锁定桌面 Chrome 是否可以接受？

### 2.4 加分项：Python、AI 字幕、WebRTC 怎么排

我当前建议：

| 加分项 | 优先级 | P0 内的边界 |
| --- | --- | --- |
| Python 高亮/语言状态回放 | 最高 | 只做高亮、语言切换、录制语言状态、回放恢复，不做执行 |
| AI 字幕 | 次优先 | P0 不做完整 ASR；如果主链路稳定，可做模拟字幕或字幕 UI |
| WebRTC | 最后 | P0 不做；P1+ 再拆成“事件实时同步”和“双向音视频面试” |

原因：

- Python 高亮成本低，对主链路侵入小，适合作为可展示加分。
- AI 字幕需要音频提取、ASR、时间戳对齐、术语纠错，容易抢主链路时间。
- WebRTC 会引入网络抖动、乱序、断线、多用户归因和双向音视频，不适合 P0。

想请老师判断：

1. 如果 P0 主链路稳定，第一加分项是否选 Python 高亮/语言回放？
2. AI 字幕是否只保留 schema 和后续方向，不在 P0 硬做？
3. WebRTC 是否明确不进入当前冲刺？
